What makes a password strong?

Not symbols. Not capital letters. Not swapping a for @. The only thing that makes a password hard to guess is entropy — the number of equally likely passwords an attacker would have to search through.

For a string of truly random characters, entropy is straightforward:

bits = length x log2(alphabet size)

20 characters from a 94-character set
  = 20 x log2(94)
  = 20 x 6.55
  ~ 131 bits

That's the number in the meter above. Every extra character multiplies the search space — which is why length beats complexity every time. A 24-character lowercase password (~113 bits) is far stronger than an 8-character one using every symbol on your keyboard (~52 bits).

What the meter does and doesn't tell you

The entropy shown here describes this generator, not the password as a string. It's honest because the characters really are drawn uniformly at random — so there's no pattern for an attacker to exploit and no dictionary to shortcut with.

That maths does not transfer to passwords you invent. Tr0ub4dor&3 looks like 11 characters of chaos, but it's a dictionary word with predictable substitutions, and a cracker that knows the trick reaches it in a fraction of the search space. Human-chosen passwords are worth a fraction of their apparent entropy. Generated ones aren't.

Rough guide: under 50 bits is weak, 75+ is strong for anything online, and 100+ is comfortably beyond brute force with foreseeable hardware.

Where the randomness comes from

This tool uses crypto.getRandomValues() — the browser's cryptographically secure random source. It never uses Math.random(), which is fast, predictable, and completely unsuitable for anything secret.

Characters are chosen with rejection sampling rather than a plain modulo. Taking random % 94 would quietly make the first few characters of the alphabet more likely than the rest, because 94 doesn't divide evenly into 232. The bias is small, but it's free to avoid, so we avoid it.

Practical advice

  • Use a password manager. The real reason people reuse passwords is that they can't remember unique ones. Generate a long random password per site and let software remember it.
  • Length first. If a site allows it, go 20+. Symbols are a distant second.
  • Turn off look-alikes only when you'll be reading the password out or typing it from a screen — it costs a little entropy for a lot of convenience.
  • Turn on 2FA anyway. A strong password doesn't help if the service leaks its database or you get phished.
  • Don't reuse. One breach somewhere becomes a breach everywhere. This is the single biggest risk, and it's unrelated to how strong any individual password is.

Generating them elsewhere

{% raw %}# Linux / macOS — 20 chars
LC_ALL=C tr -dc 'A-Za-z0-9!@#$%^&*' < /dev/urandom | head -c 20; echo

# OpenSSL
openssl rand -base64 24

# Python
python3 -c "import secrets, string; print(''.join(secrets.choice(string.ascii_letters + string.digits) for _ in range(20)))"

# Node.js
node -e "console.log(require('crypto').randomBytes(15).toString('base64url'))"

# PowerShell
-join ((48..57) + (65..90) + (97..122) | Get-Random -Count 20 | % {[char]$_}){% endraw %}

Note the Python example uses secrets, not random — the standard library's random module is a Mersenne Twister and is no safer than Math.random() for this.
